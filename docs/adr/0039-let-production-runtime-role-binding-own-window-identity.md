# Let Production Runtime Role Binding Own Window Identity

VoiceReader will define Reader Window、Playback Renderer and Playback Overlay production identities through one Production Runtime Role Binding interface. Each fixed role owns its preload source entry、preload artifact、document artifact、global exposure name and Electron web-preference policy. The build adapter、three preload entries and main-owned window adapters consume that same binding owner; production code must not infer a role from URL、pathname、renderer implementation shape or an arbitrary channel string.

The binding owner is a build-time and main-composition authority, not a renderer-selectable permission mechanism. Reader App Shell continues to own Reader Window product behavior and lifecycle; Playback Renderer and Playback Overlay retain their separate lifecycles. Each adapter may add role-specific window presentation options, but preload identity、document identity and security-relevant web preferences come from the binding. IPC endpoint ownership remains in the executable role contracts, so this decision does not add channels、payloads or business capabilities.

Build emits a deterministic `runtime-role-bindings.json` and a generated `runtime/production-runtime-role-bindings.cjs` adapter inside the exact Build Product. Both omit source paths and record only runtime role identity; the packaged main process loads the generated adapter and validates it through a side-effect-free runtime identity parser, without embedding or reopening source-owned build metadata. The Build Verifier continues to consume only the supplied `dist/`: it keeps an independent expected role matrix, verifies the JSON manifest exactly, executes the same runtime adapter loaded by main, requires both representations to agree, then executes each named preload artifact in an isolated VM and checks explicit cross-role denials. It does not import the source binding or read `src/`、build scripts、package scripts or verifier scripts. This deliberate independent duplication is an oracle, not a second production owner.

The highest source behavior seam is the Production Runtime Role Binding interface, tested as one complete three-role matrix with unsafe path、missing role、unknown role and duplicate artifact failures. Thin adapter tracers prove Reader Window、Playback Renderer and Playback Overlay consume their resolved binding. The Build Product seam separately rejects missing、extra、swapped or inconsistent runtime manifests, executes the main-loaded runtime adapter, and continues proving executable preload capability、Reader Settings/route behavior and artifact exactness. Packaged smoke remains the final real Electron assembly proof.

This decision refines ADR-0030's fixed role entrypoints and allow-list、ADR-0031's production Reader Window adapter and ADR-0032's independent Build Product verifier. All existing BrowserWindow presentation behavior、IPC contracts、HTML/CSP、Playback Session ownership、Settings、Reading History、Favorites、SQLite schema、packaging and user data remain unchanged. Rollback restores the former hand-written paths in build、main and preload entries and removes the additive manifest; no data、IPC、schema or configuration migration is involved.
